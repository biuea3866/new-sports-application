### 결함 수정 정확성

**핵심 fix (a)(b) — 정확함.**

- `SelectSeatsRequest:4` `@field:NotEmpty` 추가 → Spring Validation이 DTO 바인딩 직후 빈 배열을 차단.
- `EventApiController:65` `@Valid @RequestBody` 추가 → `MethodArgumentNotValidException` 발생.
- `GlobalExceptionHandler:42` 기존 `MethodArgumentNotValidException` 핸들러가 `ErrorStatus.UNPROCESSABLE (422)` 반환 — fix 후 422 응답 경로 완성.
- spec 허용값 `[400, 422]` 중 422 반환 → spec 준수.
- `SelectSeatsCommand.init { require(seatIds.isNotEmpty()) }` — presentation 검증이 먼저 통과하므로 이 guard는 내부 방어선으로 유지됨. 이중 검증이나 충돌 없음.

**Stock 부트스트랩 버그 수정 (c) — 결함 범위 일탈이나 정당화됨.**

수정 전 상태: `StockJpaRepository : JpaRepository<Stock, Long>, StockCustomRepository`. `StockCustomRepositoryImpl`도 `StockCustomRepository`를 `@Repository`로 구현 중 → Spring Data JPA가 `StockJpaRepository`의 custom fragment로 `StockCustomRepositoryImpl`을 자동 연결, `StockRepositoryImpl`이 `stockJpaRepository.countOutOfStockByOwnerId()`를 호출하면 실제로는 `StockCustomRepositoryImpl`에 위임됨.

`EventApiControllerTest`는 `@SpringBootTest` + Testcontainers로 전체 애플리케이션 컨텍스트를 로드한다. 위 구조는 부트스트랩 과정에서 Spring Data가 `StockCustomRepository` 빈을 중복 등록하거나 `StockRepositoryImpl`의 `stockCustomRepository` 주입이 모호해지는 부트스트랩 오류를 유발할 수 있다. 테스트 컨텍스트 로드 실패 → 재현 테스트 자체가 실행 불가 → DEF-005 재현 불가 상황이 발생하므로, 이 수정은 테스트 실행 선행 조건으로서 정당화된다.

수정 후: `StockJpaRepository`는 `JpaRepository<Stock, Long>`만 상속, `StockCustomRepository` 분리. `StockRepositoryImpl`이 `stockCustomRepository: StockCustomRepository`를 명시적으로 주입받아 `countOutOfStockByOwnerId` 호출 — Spring Data JPA Custom Repository 패턴 정석.

결론: 수정 범위 일탈이지만 fix의 재현 테스트 실행을 위해 필요한 최소 변경으로 판단. DEF-005 수정의 사전 조건 버그.

### 권고 (머지 가능하나 개선 필요)

**[권고-1]** `EventApiController.kt:22` import 순서 — `jakarta.validation.Valid`가 `org.springframework.web.bind.annotation.PostMapping` 이후, `RequestBody` 이전에 삽입됨. Kotlin/IntelliJ 기준 import 순서는 `java.*` → `javax.*`/`jakarta.*` → `kotlin.*` → 서드파티 순. `jakarta.validation.Valid`를 `jakarta.persistence.*` 그룹 또는 import 블록 상단으로 이동 권장. 동작에는 무영향이나 코드 스타일 일관성 문제.

```
// 현재 (19~23번 라인)
import org.springframework.web.bind.annotation.PostMapping
import jakarta.validation.Valid                              ← 순서 어긋남
import org.springframework.web.bind.annotation.RequestBody
```

**[권고-2]** `EventApiController.kt:75` `releaseSeats`의 `@RequestBody request: SelectSeatsRequest`에 `@Valid` 미추가. `SelectSeatsRequest`에 `@field:NotEmpty`가 부착됐으나 `releaseSeats`는 `@Valid` 없이 `SelectSeatsRequest`를 수신. 현재 DEF-005 결함 범위(selectSeats)는 수정됐으나, `releaseSeats`도 동일한 빈 seatIds 입력 시 `ReleaseSeatsCommand.init`의 require에 도달 전 검증 없이 통과할 수 있음. 동일 DTO를 쓰는 엔드포인트 일관성 차원에서 `@Valid` 추가 권장. DEF-005 fix 범위는 아니므로 별도 티켓으로 관리 가능.
