### [B-01] `StockRepositoryImpl.kt:9` — 구현체 직접 주입 (아키텍처 위반)

```kotlin
private val stockCustomRepositoryImpl: StockCustomRepositoryImpl,
```

**이유**: `be-code-convention.md` "레이어 책임" 항목에 따라 `infrastructure` 구현체끼리는 구체 클래스 직접 주입을 피하고 domain interface(`StockCustomRepository`)를 경유해야 합니다. 구현체 타입을 직접 주입하면 (a) Spring DI 교체 경로가 막히고 (b) 테스트 시 목업이 불가능해집니다.

동일한 버그를 수정한 `fix/qa-20260524-payments-me-status-500` 브랜치는 `StockCustomRepository`(interface)를 주입하는 올바른 방식을 사용하고 있습니다 (`StockRepositoryImpl.kt:7`).

```kotlin
// StockRepositoryImpl.kt
import com.sportsapp.domain.goods.StockCustomRepository

@Repository
class StockRepositoryImpl(
    private val stockJpaRepository: StockJpaRepository,
    private val stockCustomRepository: StockCustomRepository,   // ← interface 타입으로

    ...
    override fun countOutOfStockByOwnerId(ownerId: Long): Long =
        stockCustomRepository.countOutOfStockByOwnerId(ownerId)
}
```

`DashboardCountQueryTest.kt:22`도 동일하게 `StockCustomRepository`(interface) 또는 `StockCustomRepositoryImpl`(구현체) 어느 쪽을 `@Autowired`해야 하는지 재검토 필요. 테스트 고립성 관점에서 구현체 직접 주입은 허용되지만, 프로덕션 코드(`StockRepositoryImpl`)는 반드시 interface 경유 필요.

### [R-01] `TicketOrderPurchaseScenarioTest.kt:41` — `afterEach`에서 `DELETE FROM` 직접 실행

```kotlin
jdbcTemplate.execute("DELETE FROM ticket_orders")
```

`be-code-convention.md` "Soft Delete 규칙" 상 프로덕션 코드에서 hard delete는 금지이나, 테스트 픽스처 클린업은 관례적으로 허용됩니다. harness `no-sql-delete` 룰의 `file_glob`은 `*.sql`이므로 `.kt` 파일의 `jdbcTemplate` 호출은 기술적으로 차단되지 않습니다. 다만 `@Transactional` 롤백 전략(`@Rollback`)이나 `@Sql(scripts=["/sql/cleanup.sql"], ...AFTER_TEST_METHOD)` 패턴으로 전환하면 일관성이 높아집니다. 현재 구조는 기능상 문제 없음.
