### 1. `GlobalExceptionHandler.kt:47` — `exception.value` nullable 명시 처리 권고

```kotlin
detail = "Invalid value '${exception.value}' for parameter '${exception.name}'"
```

`MethodArgumentTypeMismatchException.getValue()`는 Spring 소스에서 `@Nullable Object`로 선언됩니다. Kotlin 매핑 시 `Any?`가 됩니다. 문자열 템플릿 안에서 `Any?`를 직접 보간하면 컴파일과 런타임 모두 안전하고(null이면 `"null"` 문자열로 렌더링), `!!` 위반도 아닙니다. 그러나 사용자에게 노출되는 에러 메시지에 리터럴 `"null"`이 찍히는 경우를 방지하려면 다음과 같이 명시적으로 처리하는 것이 낫습니다.

```kotlin
// 권고
detail = "Invalid value '${exception.value ?: "(unknown)"}' for parameter '${exception.name}'"
```

블로킹 이슈는 아니지만 API 응답 품질 관점에서 개선을 권고합니다.

### 2. `DashboardCountQueryTest.kt:24,129` — 테스트에서 인프라 구현체 직접 주입

```kotlin
@Autowired private val stockCustomRepositoryImpl: StockCustomRepositoryImpl,
...
val outOfStockCount = stockCustomRepositoryImpl.countOutOfStockByOwnerId(200L)
```

이미 `StockCustomRepositoryImpl`이 `@Repository`로 등록되어 있으므로 `@DataJpaTest` 또는 `BaseJpaIntegrationTest` 컨텍스트에서 autowire 자체는 동작합니다. 그러나 컨벤션상 테스트도 도메인 interface(`StockCustomRepository`)를 주입받아 사용하는 것이 인프라 구현체 교체 시 테스트 수정 비용을 줄여 줍니다.

```kotlin
// 권고
@Autowired private val stockCustomRepository: StockCustomRepository,
...
val outOfStockCount = stockCustomRepository.countOutOfStockByOwnerId(200L)
```

이번 fix의 핵심 범위(DEF-002 결함 수정)에서 벗어나므로 블로킹하지 않으나, 후속 리팩토링 티켓으로 처리를 권고합니다.
