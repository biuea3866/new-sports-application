# DEF-001 Fix Review

## 변경 요약
`POST /goods-orders` 멱등 키 중복 요청 시 신규 주문 생성 방지 — `idempotency_key` 컬럼 추가(V23) + `GoodsDomainService.createPendingOrder` 가드 + `CreateGoodsOrderUseCase` 분기 처리 + [E2E-06-R02] 시나리오 테스트 추가.

## verdict
request-changes

## Must Fix (블로커)

### 1. `CreateGoodsOrderUseCase` — `@Transactional` 누락 (be-code-convention.md "기본 트랜잭션: UseCase @Transactional")
- **`CreateGoodsOrderUseCase.kt:14`** `@Service` 선언만 있고 `@Transactional`이 없음
- `execute()`는 여러 DomainService를 오케스트레이션하며 `cancelPendingOrder` / `markPaid` 를 포함한 다단계 쓰기 연산을 수행한다. 트랜잭션 없이 중간 단계 실패 시 데이터 정합성이 깨진다.
- 수정 방향: `@Transactional` 을 `execute()` 에 추가.

### 2. `CreateGoodsOrderUseCase.execute()` — 10줄 초과 (be-code-convention.md "UseCase execute() 10줄 이내")
- **`CreateGoodsOrderUseCase.kt:22-45`** `execute()` 본문이 24줄 (시그니처·닫는 중괄호 포함). 실질 로직 줄만 세어도 12줄 초과.
- 수정 방향: `buildIdempotentResponse` 분기와 결제 흐름을 DomainService에 위임하거나 private 헬퍼 추출 후 `execute()`를 10줄 이하로 축소.

### 3. Race condition — 동시 요청 시 `DataIntegrityViolationException` 500 응답 (보안·성능)
- **`GoodsDomainService.kt:60`** `goodsOrderRepository.findByIdempotencyKey(idempotencyKey)?.let { return it }` 이후 `goodsOrderRepository.save(...)` 까지 트랜잭션 격리·비관적 락이 없음.
- 동일 키로 두 요청이 동시에 `findByIdempotencyKey`를 통과하면 둘 다 `save`를 시도하고, DB unique constraint(`uq_goods_orders_idempotency_key`)에서 후행 요청이 `DataIntegrityViolationException`을 발생시켜 500 반환.
- 정상 멱등 응답(202)을 기대하는 클라이언트에게 500이 반환되므로 재시도 루프를 유발한다.
- 수정 방향 (둘 중 하나): (a) `createPendingOrder` 에 `@Transactional` + `SELECT ... FOR UPDATE` 비관적 락으로 직렬화, (b) `DataIntegrityViolationException` catch 후 기존 row를 재조회해 반환하는 "insert-or-fetch" 패턴 적용.

### 4. `StockRepositoryImpl` — 구체 클래스(`StockCustomRepositoryImpl`) 직접 주입 (아키텍처 레이어 위반)
- **`StockRepositoryImpl.kt:10`** `private val stockCustomRepositoryImpl: StockCustomRepositoryImpl`
- `StockRepositoryImpl`(`infrastructure`)이 `StockCustomRepositoryImpl`(`infrastructure`) 구체 클래스를 직접 주입. 의존 방향상 interface(`StockCustomRepository` — `domain`)를 주입받아야 하며, Spring이 `StockCustomRepositoryImpl`을 자동 주입한다.
- 수정 방향: `private val stockCustomRepository: StockCustomRepository`로 교체.

## Nice to Have

### N-1. 멱등 응답에서 `paymentId` null 케이스 — 클라이언트 계약 불명확
- **`CreateGoodsOrderUseCase.kt:49`** `buildIdempotentResponse` 가 `OrderWithPayment(paymentId = order.paymentId)` 를 반환. 기존 주문이 아직 PENDING 상태이면 `paymentId = null`, `paymentStatus = null` 로 응답한다. 클라이언트가 이 케이스를 처리하는지 확인 필요.
- 방향: API 문서·클라이언트 계약에 `paymentId` nullable 케이스를 명시.

### N-2. `processPaymentResult` `else` 브랜치 — `cancelPendingOrder` 호출로 데이터 손실 가능
- **`CreateGoodsOrderUseCase.kt:58-61`** `PaymentStatus.COMPLETED`·`FAILED` 외 상태(`PENDING`, `IN_PROGRESS` 등)에서 `cancelPendingOrder`를 호출. 비동기 결제 시스템에서 `PENDING` 상태로 콜백이 오면 주문이 의도치 않게 취소된다.
- 방향: `else` 케이스는 취소 대신 로깅 후 `PENDING` 유지 또는 별도 보상 플로우로 처리.

### N-3. GoodsOrderScenarioTest `beforeEach` — `DELETE FROM` 직접 실행
- **`GoodsOrderScenarioTest.kt:40-44`, `163-167`** `jdbcTemplate.execute("DELETE FROM ...")` 를 직접 사용. harness `no-sql-delete` 룰의 file_glob 이 `*.sql` 이므로 `.kt` 파일은 차단 대상 외이지만, 테스트 격리는 `@Sql`(DDL 롤백) 또는 `@Transactional`(롤백) 방식이 더 안전하다.
