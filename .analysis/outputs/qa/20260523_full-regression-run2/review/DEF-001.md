## 필수 수정 (블로커)

### 1. root cause 미해결 — 두 번째 401 body 출처 미확인

**근거 (코드 트레이스 결과)**

`SecurityConfig.kt`의 `jsonAuthenticationEntryPoint`(라인 66-71)는 `{"status":401,"title":"Unauthorized","detail":"Authentication required"}` 형식을 반환합니다. 그러나 DEF-001 결함 md에는 두 번째 body `{"error":"unauthorized","message":"missing or invalid bearer token"}` 가 LOAD-01 1,459건에서 수신됐다고 기록됩니다.

be-implementer는 "BE 코드 전체 grep에서 미검출 → 환경 문제 추정"으로 결론지었습니다. 그러나 다음 후보를 직접 검증하지 않았습니다.

- `JwtAuthenticationFilter.doFilterInternal` (라인 21-41): 토큰 없는 경우 SecurityContext 미설정 후 `filterChain.doFilter()` chain-through. 401 직접 반환 없음 — 이 부분은 확인됨.
- `OwnershipGuardImpl.authUserId()` (라인 19-22): `SecurityContextHolder.getContext().authentication?.principal as? UserPrincipal ?: throw UnauthorizedException()`. 이 경로는 `permitAll()` 경로여도 UseCase가 `ownershipGuard.authUserId()`를 호출하면 진입 가능. `UnauthorizedException` → `GlobalExceptionHandler.handleBusinessException` → `{"status":401,...}` 형식이므로 body 형식은 다름.
- QA 환경 `docker-compose.qa.yml`: nginx/proxy 레이어 없음 확인. BE가 직접 포트 8080 노출.
- `application.yml:1` — `server.servlet.context-path` 미설정, `server.port: 8080` 확인.

**미검증 항목**: k6 `facility-search.js` 에서 실제로 어떤 URL로 요청했는지, `QA_API_URL` 환경변수 값이 무엇이었는지. `auth.js` 헬퍼(DEF-010 수정본)에서 `headerAuth(userId)` 방식이 실제 부하 시나리오에서 올바르게 적용됐는지.

두 번째 body가 qa-load-tester 환각이라는 결론은 타당하지만, 직접 아티팩트(raw.json/raw.log)로 검증하지 않았습니다. COMPLETION-RULE §2 "테스트 통과 단언은 도구 호출 아티팩트가 선행돼야 한다"에 해당합니다.

**요구 조치**: `load-results/facility-search/raw.json` 또는 `raw.log`를 직접 열어 실제 response body를 확인하고, 환각 여부를 아티팩트로 입증하거나 실제 두 번째 출처를 식별해야 합니다. 아티팩트 없이 "환경 문제로 추정"으로 종결하면 Step 5-B 재검증에서 동일 결함 재발 시 원인 추적 불가합니다.

### 3. auth/login 케이스 단언이 Security 필터 차단을 감지하지 못함

**파일:라인** `SecurityConfigIntegrationTest.kt:75-84`

```kotlin
Then("401 이상의 도메인 예외가 반환되고 Spring Security 필터 레벨 차단이 아니어야 한다") {
    response.statusCode shouldNotBe HttpStatus.FORBIDDEN
    val statusCode = response.statusCode.value()
    statusCode shouldBeGreaterThanOrEqualTo 400
}
```

이 단언은 `shouldBeGreaterThanOrEqualTo 400`이므로 Security 필터가 /auth/login을 차단해 401을 반환해도 통과합니다. DEF-001 결함의 재현 목적(permitAll 미적용 시 401 감지)을 달성하지 못합니다.

잘못된 자격증명으로 인한 도메인 401(`UnauthorizedException`, body: `{"status":401,...}`)과 Security 필터 401(body: `{"status":401,"title":"Unauthorized","detail":"Authentication required"}`)은 body로만 구분 가능합니다. status code만으로는 두 경우를 분리할 수 없습니다.

**요구 조치**: body에서 `Content-Type` 또는 특정 필드(`"title"` 존재 여부)를 검사하는 단언 추가, 또는 `/auth/login`을 올바른 자격증명으로 호출해 200을 단언하는 방식으로 변경.

### 4. 아키텍처 위반 발견 — UseCase에서 OwnershipGuard 직접 주입 (기존 코드, PR 외 범위)

PR #131에서 도입한 코드는 아니지만, 리뷰 중 발견한 위반입니다. be-code-convention.md "UseCase 규칙" 위반이며 PR 범위가 아니므로 별도 티켓으로 등록 권장합니다.

- `application/ticketing/GetMyEventWithSalesUseCase.kt`, `OpenMyEventUseCase.kt`, `CloseMyEventUseCase.kt` 동일.

`OwnershipGuard`는 domain interface이나, `authUserId()`가 SecurityContext를 직접 의존하는 infrastructure 구현체(`OwnershipGuardImpl`)를 통해 주입됩니다. UseCase는 DomainService만 주입받아야 하므로, `authUserId()` 획득은 presentation 레이어(Controller)에서 수행 후 Command에 담아 전달하거나, DomainService가 OwnershipGuard를 주입받아 처리하는 방향이 맞습니다.

이 위반이 허용된 배경(TODO: AUTH-04 주석)은 인지하나, 현재 구조는 UseCase가 SecurityContext 의존성을 갖는 문제가 있습니다.

## 머지 가능 여부

**현재 상태: 머지 불가**

이유:
1. TDD RED→GREEN 검증 아티팩트 없음 — 테스트가 실제로 통과하는지 미확인.
2. 두 번째 401 body 출처가 아티팩트로 입증되지 않아, "환경 문제 결론"의 근거가 부족함.
3. `auth/login` 케이스 단언이 DEF-001 재현 목적을 달성하지 못함.

위 3개 필수 수정 항목을 해소한 뒤 재검토 가능.

단, **환경 문제라는 진단 방향 자체는 코드 트레이스 상 합리적**입니다. BE 코드에 두 번째 401 생성 경로가 없음이 grep으로 확인됐고, SecurityConfig의 permitAll 선언이 코드상 올바릅니다. raw.json 검증에서 환각이 입증되면 테스트 케이스 수정(블로커 3)과 CI 통과 확인(블로커 2)만으로 재승인 가능합니다.
